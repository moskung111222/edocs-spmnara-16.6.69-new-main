# Business Rules

ตรรกะทางธุรกิจ ข้อบังคับ และเงื่อนไขการทำงานหลักของระบบ NWT Document Submission System

---

## 1. สถานะและการไหลของขั้นตอนการทำงาน (Status Flow)

สถานะการดำเนินงานของคำขอมีอยู่ทั้งหมด 8 สถานะ โดยสามารถแสดงลำดับขั้นการทำงานดังนี้:

```
[submitted] ──> [received] ──> [in_review] ──> [pending_approval] ──> [approved] / [completed]
                                    │
                                    ├──> [need_info] ──> (ผู้ใช้ส่งหลักฐานแก้) ──> [submitted]
                                    │
                                    └──> [rejected]
```

1. **submitted (ยื่นคำขอแล้ว)**: คำขอส่งเข้ามาในระบบหลังผ่านการยืนยัน OTP เรียบร้อย
2. **received (รับเรื่องแล้ว)**: เจ้าหน้าที่กดรับเรื่อง เพื่อระบุว่าเอกสารอยู่ในคิวเตรียมตรวจสอบ
3. **in_review (กำลังตรวจสอบเอกสาร)**: เจ้าหน้าที่เริ่มตรวจไฟล์แนบ และแบบฟอร์มประวัติการศึกษา
4. **need_info (ขอข้อมูลเพิ่มเติม/แก้ไขเอกสาร)**: เจ้าหน้าที่ตรวจพบไฟล์ชำรุด/ไม่ชัดเจน และร้องขอให้ประชาชนอัปโหลดใหม่ ประชาชนจะได้รับเมลชี้แจง และปุ่มอัปโหลดเพิ่มจะเปิดขึ้นบนหน้าจอติดตามสถานะ เมื่อประชาชนอัปโหลดแก้ไขสำเร็จ สถานะจะเด้งกลับไปที่ `submitted` โดยอัตโนมัติ
5. **pending_approval (รอการอนุมัติ)**: เอกสารถูกต้องครบถ้วน เจ้าหน้าที่เสนอเรื่องต่อหัวหน้ากลุ่มงาน
6. **approved (อนุมัติคำขอแล้ว)**: หัวหน้ากลุ่มอนุมัติคำขอสำเร็จ
7. **completed (ดำเนินการเสร็จสิ้น (รับเอกสาร))**: ประชาชนเดินทางมารับใบคำขอตัวจริง หรือระบบส่งเอกสารสำเร็จ
8. **rejected (ปฏิเสธคำขอ)**: คำขอถูกยกเลิก (เนื่องจากข้อมูลขัดแย้งกับทะเบียนประวัติ หรือไม่ใช่สิทธิ์)

---

## 2. กฎการอนุมัติเอกสาร (Approval Rules)

- เฉพาะผู้ใช้งานสิทธิ์ **Head (หัวหน้ากลุ่มงาน)** และ **Admin (ผู้ดูแลระบบ)** เท่านั้นที่สามารถเปลี่ยนสถานะคำขอให้เป็น `approved` หรือ `completed` ได้
- หากผู้ใช้สิทธิ์ **Staff (เจ้าหน้าที่)** พยายามเลือกอนุมัติ ระบบจะบล็อกคำสั่ง และโยนข้อผิดพลาดเตือนสิทธิ์ (สิทธิ์ไม่ถึง) ผ่าน **RequestService::changeStatus()**
- ในกรณีที่เปลี่ยนสถานะเป็น `need_info` หรือ `rejected` บังคับให้เจ้าหน้าที่ต้องกรอกเหตุผล/คำชี้แจงทุกครั้ง เพื่อเป็นประวัติแจ้งเตือนประชาชนผ่านอีเมล

---

## 3. เงื่อนไขการยืนยันตัวตนด้วย OTP (OTP Rules)

- รหัส OTP ประกอบด้วยตัวเลขจำนวน 6 หลักแบบสุ่ม
- รหัสผ่านมีเวลาหมดอายุการใช้งาน **5 นาที (300 วินาที)** นับแต่วันเวลาที่บันทึก
- จำกัดความพยายามในการกรอกรหัสผิดพลาดได้ไม่เกิน **5 ครั้ง** หากเกินจำนวนสิทธิ์ รหัส OTP บล็อกการยืนยัน
- **Dev/Test Mode**: ในระหว่างพัฒนาและทดสอบระบบในสภาพแวดล้อมจำลอง (localhost) ระบบจะยินยอมรหัสผ่าน Master OTP `123456` ให้ผ่านการตรวจสอบได้ตลอดเวลาเพื่อความรวดเร็ว

---

## 4. เงื่อนไขและประเภทการแนบเอกสาร (Upload Rules)

- **ประเภทไฟล์**: ยินยอมแนบเฉพาะไฟล์เอกสารรูปแบบ **PDF (`application/pdf`)** เท่านั้น
- **ขนาดไฟล์สูงสุด**: ไม่เกิน **10 MB** (`10 * 1024 * 1024` ไบต์) ต่อหนึ่งไฟล์
- **การตรวจสอบความถูกต้อง**:
  - ตรวจเช็ก MIME Type จากโครงสร้างภายในด้วย `finfo_file`
  - ตรวจสอบความถูกต้องของโครงสร้างหัวไฟล์ PDF (Magic Bytes) ต้องมีตัวอักษรเริ่มต้นที่พิกัด 0–3 ตรงกับคำว่า `%PDF`
- **ระบบชื่อไฟล์**: ระบบจะทำการสุ่มเปลี่ยนชื่อไฟล์ทั้งหมดโดยใช้รหัสแฮช 16 ไบต์ (`bin2hex(random_bytes(16)) . '.pdf'`) เพื่อป้องกันการทับซ้อนและหลบเลี่ยงความปลอดภัยชื่อไฟล์ดั้งเดิม
- **การจัดการเวอร์ชันหลักฐาน (Versioning)**: หากผู้ยื่นคำขอส่งไฟล์แนบรายการเดิมซ้ำเข้ามาหลายรอบ (เช่นกรณีแก้ไขข้อมูลเพิ่มเติม) ระบบจะเพิ่มเลขเวอร์ชันขึ้นทีละขั้น (v1, v2, v3, ...) ในตารางข้อมูล เพื่อเก็บรักษาร่องรอยและเอกสารประวัติเก่าไว้ตรวจสอบย้อนหลัง
